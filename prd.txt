# 📊 Project Overview

DB-Lighthouse AI is an open-source developer tool that connects to a PostgreSQL database or SQL repository and provides:

* Live relationship visualization
* Storage optimization analysis
* Indexing recommendations
* AI-assisted schema design

The MVP focuses on **cost reduction and performance optimization** using intelligent heuristics + local LLM reasoning (Jan AI).

Target deployment: **Local demo app (Dockerized)**
Target users: **Backend developers in startups**

---

# 🎯 Product Vision

Backend developers struggle with:

* Bloated schemas increasing cloud DB cost
* Poor indexing causing slow queries
* Hard-to-understand table relationships
* Outdated ER diagrams

DB-Lighthouse acts as a **read-only observability + optimization assistant** that:

1. Connects to a live PostgreSQL DB
2. Creates a shadow clone
3. Analyzes schema + metadata
4. Suggests measurable storage & performance improvements
5. Visualizes everything clearly

Primary Value Proposition:

> “Reduce your database storage footprint and improve query performance with AI-driven insights — safely.”

# ✨ Core Features (MVP – Exact Specs)

## 1️⃣ Database Connection & Shadow Clone

**Supported Inputs:**

* Live PostgreSQL connection string
* Uploaded `.sql` file
* GitHub repo containing `.sql` files

**Behavior:**

* User enters credentials
* System creates read-only connection
* Shadow schema is cloned into local PostgreSQL container
* No destructive operations allowed on original DB
* Credentials stored in memory only (session-based)

---

## 2️⃣ Live Relationship Mapping (2D with Depth Effects)

Built using React Flow + layered shadows.

### Functional Specs:

* Nodes represent tables
* Edges represent foreign keys
* Thickness of edge = frequency of reference
* Node size = row count estimate
* Node color heatmap:

  * Green → Low storage
  * Yellow → Medium
  * Red → High storage

### Features:

* Hover → Show:

  * Table name
  * Row count
  * Estimated size
  * # of foreign keys
* Click → Right panel opens:

  * Column breakdown
  * Index list
  * Storage estimate
* “Highlight Bottlenecks” toggle:

  * Flags tables with >5 foreign key connections

No 3D engine. Simulated depth via shadow + scaling.

---

## 3️⃣ Space Shrinker (Core Feature – Cost Optimization Engine)

Primary success metric driver.

### Analysis Logic:

For each table:

* Detect TEXT columns with avg length < 255 → suggest VARCHAR
* Detect VARCHAR(1000) with avg < 100 → suggest smaller size
* Detect unused indexes
* Detect redundant indexes
* Detect nullable columns with 0 NULL values
* Detect repeated enum-like values → suggest normalization

### Data Sampling:

* SELECT * LIMIT 1000 rows
* Compute:

  * avg column length
  * max column length
  * null %
  * distinct value count

### Output:

Optimization Report Card:

| Metric                  | Before | After | Savings |
| ----------------------- | ------ | ----- | ------- |
| Total Estimated Storage | 1.2GB  | 950MB | 20.8%   |

Each suggestion includes:

* Risk level (Low/Medium/High)
* Generated SQL patch
* “Apply to Shadow DB” button

System asks confirmation before applying.

---

## 4️⃣ Predictive Indexing (Rule-Based + LLM)

Not true RL (MVP simplification).

### Inputs:

* pg_stat_statements
* EXPLAIN ANALYZE output
* Query frequency logs

### Heuristics:

* Sequential scan on large table → suggest index
* Frequent WHERE on non-indexed column → suggest index
* Multi-column filter → suggest composite index
* ORDER BY without index → suggest index

LLM Role:

* Explain why index is suggested
* Rank impact (High/Medium/Low)

Output example:

“Adding index on users(email) could reduce average query time by 65%.”

SQL patch generated but not auto-applied.

---

## 5️⃣ Natural Language → Schema Generator

User enters:

> “Food delivery app with users, restaurants, orders, delivery agents.”

System generates:

* ER diagram
* Normalized SQL schema (3NF)
* Downloadable .sql file

No live DB required for this module.

# 🔄 Key User Flows

## Flow 1: Optimize Existing DB

1. Login
2. Enter DB credentials
3. Shadow clone created
4. System scans schema (5–15 sec)
5. Dashboard displays:

   * Total size
   * Risk tables
6. User clicks “Run Optimization Scan”
7. Space Shrinker generates report
8. User reviews suggestions
9. Click “Apply to Shadow”
10. System recalculates size
11. Shows % reduction

---

## Flow 2: Analyze Performance

1. Open Performance Tab
2. System reads query stats
3. Detects missing indexes
4. Shows ranked suggestions
5. User applies to shadow DB
6. EXPLAIN re-run
7. Show before/after latency

---

## Flow 3: Generate New Schema

1. Open Schema Builder
2. Enter description
3. LLM generates schema
4. Visual ER displayed
5. Export SQL
