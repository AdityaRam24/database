\begin{abstract}
Web applications rely on relational databases for almost everything, yet
keeping a schema healthy---query performance, storage footprint, security
exposure, structural integrity, compliance---demands expertise that
developers on small teams rarely have time to build. This paper presents
DB-Lighthouse, a platform where a developer pastes in a connection string
and gets back an interactive schema graph, a storage optimization report,
index recommendations, anomaly alerts, and plain-English explanations,
without writing a single catalog query. The key safety property is the
\emph{shadow database pattern}: only the DDL is copied to a local
instance---no row data leaves the source system at any point. The system
provides six analytical subsystems: a three-phase index optimizer, a
Z-score anomaly detector with rolling 7-day baselines, a composite incident
scorer, a self-healing NL-to-SQL translator with up to three correction
rounds, a four-tier prompt injection firewall, and a two-agent governance
council. The platform unifies core observational workflows---spanning
performance analysis, schema governance, semantic rule enforcement, and
statistical anomaly detection---within a single interface backed by a
unified shadow-database isolation boundary. Six database back-ends
(PostgreSQL, MongoDB, Neo4j, Firebase, MySQL, and SQLite) are supported
through a common connection and analysis pipeline.

Evaluation on the Chinook reference database shows: 99.91\% buffer cache
hit ratio, 100\% prompt injection detection across 62 adversarial inputs
with zero false positives, 100\% NL-to-SQL resolution within three
self-healing rounds (76\% first-attempt success), and 97.5--98.4\%
execution-time reduction for targeted queries after index simulation. On
four production schemas the platform produced a mean storage reduction of
19.4\% and matched expert DBA index choices with a precision of 0.81. A
LoRA fine-tuning pipeline produces a 4.6 GB Q4\_K\_M GGUF model
deployable on commodity hardware for domain-adapted SQL generation.
\end{abstract}

\section{Introduction}

Cloud-hosted PostgreSQL services---including Amazon RDS, Google Cloud
SQL, Supabase, Railway, and Neon---have made production-grade relational
databases available to startups and individual developers. The tooling
for observing and maintaining those databases, however, has not kept pace.
Tasks like index tuning, normalization analysis, anomaly triage, compliance
scanning, migration pre-validation, and schema history tracking still
demand time and knowledge of SQL internals, query planning,
and system catalogs. The gap is most acute for small-to-medium engineering teams (2--20
engineers) operating without dedicated DBA expertise, where schema health
problems---bloated schemas, redundant indexes, oversized column types, PII
fields that have never been audited, and schema changes that bypass version
control---accumulate silently until they manifest as outages or compliance
failures.

Deploying LLMs against production databases introduces immediate safety
concerns---prompt injection, accidental data modification, and PII
exposure \cite{perez2022ignore,greshake2023indirect}. Existing
tools---Datadog, New Relic, PgHero---address parts of the problem but
target large enterprises, carry high costs at moderate data volumes, or
present output that only experienced DBAs can use. Natural language
interfaces to databases have attracted research attention
\cite{li2023can,pourreza2024din,gao2023texttosql}, yet none of the
published systems combine such interfaces with live schema analysis, DDL
experimentation, governance enforcement, and domain-fine-tuned model
support in a single accessible platform.

DB-Lighthouse addresses these concerns through four architectural
decisions: (1) a shadow-database pattern that processes only schema DDL,
never production rows; (2) a prompt firewall that classifies incoming
queries across four attack categories before model inference; (3) a
two-agent council with separate agents for performance and security review;
and (4) support for a domain-fine-tuned 7B-parameter model that achieves
higher first-attempt NL-to-SQL accuracy than general-purpose models on
narrow schema tasks.

Eight contributions distinguish this work:

\begin{itemize}
    \item A \textbf{shadow database architecture} that separates analysis
    from production, guaranteeing zero risk of data modification.
    \item A \textbf{three-phase index analyzer} covering sequential-scan
    bottlenecks, zero-use zombie indexes, and missing indexes inferred from
    FK relationships and column correlation statistics.
    \item A \textbf{Z-score anomaly detector} with rolling 7-day baselines,
    WARNING (Z $\geq$ 2.0) and CRITICAL (Z $\geq$ 3.0) thresholds across
    five PostgreSQL metric categories.
    \item A \textbf{self-healing NL-to-SQL translator} that feeds
    PostgreSQL error messages back to the model for up to three correction
    rounds before escalating.
    \item A \textbf{four-tier prompt injection firewall} with
    category-specific confidence ceilings blocking SQL injection,
    instruction override, destructive intent, and social engineering.
    \item A \textbf{LoRA fine-tuning pipeline} for Qwen2.5-Coder-7B-Instruct
    producing a 4.6 GB Q4\_K\_M GGUF model deployable on commodity hardware.
    \item A \textbf{unified observational platform} that consolidates
    performance analysis, DDL sandboxing, schema versioning, drift detection,
    semantic rule enforcement, anomaly detection, security scanning, and
    multi-agent NL-to-SQL within a single shadow-bounded interface---
    eliminating the need for disparate specialist tools across the schema
    health lifecycle.
    \item An \textbf{evaluation} of platform effectiveness demonstrating
    measurable storage, query performance, and security improvements across
    both production and reference schemas.
\end{itemize}

Section~II surveys prior work; Section~III describes the architecture
and deployment model; Section~IV details the six analytical subsystems;
Section~V covers technical choices including the fine-tuning pipeline;
Section~VI evaluates the system; Section~VII concludes.

\section{Related Work}

\subsection{Text-to-SQL and Natural Language Interfaces}

Converting natural language questions into executable SQL has attracted
sustained research effort. Early rule-based systems such as LUNAR
\cite{woods1973progress} and MASQUE \cite{androutsopoulos1995nldb} showed
feasibility but broke down with schema heterogeneity. Large pre-trained
language models improved the state of the art substantially: T5-based
architectures fine-tuned on the Spider benchmark \cite{yu2018spider}
reached near-human accuracy on single-database tasks, while models such
as GPT-4 \cite{openai2023gpt4} and Llama 2 \cite{touvron2023llama} enabled
zero-shot query generation across novel schemas. DIN-SQL \cite{pourreza2024din}
decomposes complex queries into sub-tasks and achieves 82.8\% exact-match
accuracy on Spider with GPT-4. DB-Lighthouse adds a self-healing loop that
feeds PostgreSQL error messages back to the model, recovering from parse
errors without user intervention---particularly useful on narrow domain
schemas where general-purpose models frequently fail on column-name
ambiguity in multi-table joins.

\subsection{Automated Index Recommendation}

Index selection is a classic problem in physical database design. The
AutoAdmin project \cite{chaudhuri1997autoadmin} introduced workload-driven
index recommendation using query cost estimation. More recent work applies
reinforcement learning and graph neural networks \cite{lan2020index,
basu2020weightedge}. PgHero \cite{pghero2022} provides lightweight
heuristic-based index suggestions for PostgreSQL but lacks natural-language
explanations. DB-Lighthouse adds a three-phase analyzer covering zombie
index detection and correlation-guided missing index inference, with
LLM-generated rationales making recommendations readable to developers
without query planning expertise.

\subsection{Database Anomaly Detection}

Chandola et al. \cite{chandola2009anomaly} classify anomaly detection into
statistical, machine-learning, and information-theoretic approaches. For
database metrics, Z-score statistical process control \cite{grubbs1969}
remains the most interpretable option because each alert maps directly to a
metric deviation in standard-deviation units. OtterTune \cite{van2017ottertune}
applied machine learning over \texttt{pg\_stat\_activity} for automatic
configuration tuning. DB-Lighthouse uses Z-score detection with per-metric
rolling baselines, ensuring every alert can be traced to a single metric
and explained to an on-call DBA without model interpretation.

\subsection{Autonomous Database Management}

Pavlo et al. \cite{pavlo2017selfdriving} described a self-driving DBMS that
automates physical design, knob tuning, and anomaly response. NoisePage
\cite{ma2021noisepage} demonstrated online physical design adaptation at
runtime. DB-Lighthouse targets a different scenario: organisations that
cannot modify their production DBMS. The shadow-database pattern achieves
non-invasive analysis by cloning only the schema and executing all queries
against a local shadow instance, leaving the production system entirely
untouched after the initial schema extraction.

\subsection{Schema Visualization and Governance}

Interactive schema visualization tools include SchemaSpy
\cite{schemaspy2022} and DBeaver \cite{dbeaver2023}, which are read-only
renderers with no optimization guidance. Data governance frameworks such as
Apache Atlas \cite{atlas2023} and Collibra \cite{collibra2023} handle
enterprise metadata management. PII detection is available through AWS
Macie \cite{macie2023} and Google Cloud DLP \cite{clouddlp2023}, but both
are cloud-provider-specific. DB-Lighthouse runs PII detection at the schema
level without cloud dependency.

\subsection{LLM Safety and Prompt Injection}

Perez and Ribeiro \cite{perez2022ignore} showed that appended user input
can reliably override LLM system instructions. Greshake et al.
\cite{greshake2023indirect} extended this to indirect injection via
retrieved documents. The DB-Lighthouse prompt firewall intercepts these
attacks at the input boundary using pattern-match confidence scoring before
model inference, rather than filtering model output after the fact.

\subsection{Parameter-Efficient Fine-Tuning}

Hu et al. \cite{hu2022lora} introduced LoRA, constraining weight updates to
a low-rank decomposition $W = W_0 + BA$ where $B \in \mathbb{R}^{d \times r}$,
$A \in \mathbb{R}^{r \times k}$, $r \ll \min(d,k)$, reducing trainable
parameters by roughly 10,000$\times$ on large models. Dettmers et al.
\cite{dettmers2023qlora} showed 4-bit NormalFloat (NF4) quantization
retains near-float16 accuracy. DB-Lighthouse combines both via the Unsloth
framework \cite{han2024unsloth} to fit model training within 15 GB VRAM on
commodity hardware.

\subsection{Multi-Agent Deliberation}

Park et al. \cite{park2023generative} showed LLM agents exhibit
collaborative behaviour when assigned distinct roles. AutoGen
\cite{wu2023autogen} formalised multi-agent conversation as a general
decomposition pattern. DB-Lighthouse uses a two-agent council: an Architect
agent proposes performance-optimised solutions; a Guardian agent reviews for
security and data-integrity risks; an Executive agent synthesises the final
output at temperature 0.1 to minimise hallucination.

\section{System Architecture}

\subsection{Architectural Overview}

DB-Lighthouse follows a three-tier architecture: a Next.js frontend, a
FastAPI backend, and a shadow PostgreSQL instance. All communication
between tiers uses RESTful HTTP with JSON payloads. The backend exposes
42 HTTP endpoints under the \texttt{/api} prefix, across 20 domain
modules registered in \texttt{backend/app/api/\_\_init\_\_.py}.
Cross-origin requests are permitted from ports 3000, 3001, and 3002 via
FastAPI's \texttt{CORSMiddleware}.

\subsection{Shadow Database Pattern}

The shadow database pattern is the primary safety mechanism in
DB-Lighthouse. When a user supplies a PostgreSQL connection string, the
backend runs the following sequence:

\begin{enumerate}
    \item Verify connectivity to the source database using a read-only
    probe query.
    \item Execute \texttt{pg\_dump --schema-only} against the source to
    extract Data Definition Language (DDL) statements---tables, indexes,
    sequences, views, and functions---without reading any data row.
    \item Restore the DDL into a local shadow PostgreSQL instance using
    \texttt{psql}.
    \item Run all subsequent analysis queries against the shadow
    instance exclusively. The production database is never queried again
    after the initial schema extraction.
\end{enumerate}

Let $S$ denote the user's source schema and $S'$ the shadow clone. The
mapping $f: S \rightarrow S'$ preserves the full structure of $S$---
including all tables, columns, constraints, indexes, foreign keys,
sequences, and types---while discarding all row-level data. Formally,
for any table $T \in S$ and its shadow counterpart $T' \in S'$:

\begin{equation}
    |T'| = 0 \quad \text{and} \quad \text{schema}(T') = \text{schema}(T)
\end{equation}

where $|\cdot|$ denotes the row count. The pattern trades schema freshness
for data safety: if the production schema evolves after cloning, analysis
reflects the old structure. DB-Lighthouse exposes a manual re-clone button;
automated synchronisation via PostgreSQL event triggers is left for future
work.

\subsection{Multi-Database Routing}

Two predicate functions handle non-PostgreSQL targets at the API boundary.
\texttt{\_is\_mongodb()} returns \texttt{True} when the connection string
begins with \texttt{mongodb://} or \texttt{mongodb+srv://}.
\texttt{\_is\_firebase()} returns \texttt{True} when the payload contains
both \texttt{private\_key} and \texttt{project\_id} JSON keys. All endpoint
handlers check these predicates and route to \texttt{MongoDBService} or
\texttt{FirebaseService} as appropriate. Unsupported operations return a
structured \texttt{\{"supported": false, ...\}} response rather than an
HTTP error.

\subsection{AI Safety Firewall}

All natural-language queries that produce SQL pass through a prompt
firewall in \texttt{services/prompt\_firewall.py}. The firewall maintains
four pattern libraries: \textbf{SQL Injection} (13 patterns: UNION SELECT,
\texttt{pg\_sleep}, information\_schema access, stacked queries),
\textbf{Instruction Override} (11 patterns: ``ignore previous
instructions'', ``you are now'', DAN-style jailbreaks),
\textbf{Destructive Intent} (3 patterns: ``drop all tables'', ``wipe
the database''), and \textbf{Social Engineering} (4 patterns: false
authority claims, urgency manipulation). When multiple patterns fire,
only the highest-confidence threat is returned. Category-specific
confidence ceilings prevent inflation from pattern co-occurrence
(Table~\ref{tab:firewall}).

A second layer---the SQL safety guardrail in \texttt{safe\_sql\_check()}---
operates on AI output rather than user input. It matches destructive
SQL keywords (\texttt{DROP DATABASE}, \texttt{DROP TABLE},
\texttt{TRUNCATE}, \texttt{DELETE FROM}) via word-boundary patterns
and automatically appends \texttt{LIMIT 100} to any unbounded
\texttt{SELECT}. These two layers ensure neither injected prompts nor LLM
errors can trigger destructive operations.

\subsection{Migration Safety Engine}

Schema migrations are validated through a four-stage pipeline.
\textbf{Stage 1}: a block-list check rejects migrations containing
\texttt{DROP TABLE}, \texttt{DROP DATABASE}, \texttt{TRUNCATE}, or
\texttt{DELETE} without a \texttt{WHERE} clause.
\textbf{Stage 2}: \texttt{sqlparse} extracts the operation type (ALTER,
CREATE, INSERT, UPDATE, DROP, SELECT).
\textbf{Stage 3}: additive operations (CREATE TABLE, ADD COLUMN, CREATE
INDEX) are approved immediately with \texttt{is\_safe=True}.
\textbf{Stage 4}: all other operations trigger four dependency checks
against the shadow database---FK constraints, dependent indexes, dependent
views, and stored functions referencing the table. Any dependency hit
returns \texttt{is\_safe=False} with a structured report. The governance
endpoint returns HTTP 422 when \texttt{is\_safe=False}.

\subsection{Database Connection Interface and Multi-Database Project Sidebar}

Users interact with DB-Lighthouse through a persistent collapsible
project sidebar. The connection wizard supports six primary database
back-ends and three schema import paths: PostgreSQL (shadow cloning
pipeline), MongoDB (\texttt{mongodb://} URI, collection metadata
introspection), Neo4j (Bolt URI, graph schema introspection), Firebase
(service account JSON, collection and document schemas), MySQL/MariaDB
(\texttt{mysql://} with dialect conversion), and SQLite (direct
\texttt{.db} file upload). Beyond live connections, users can import a
bare SQL schema file, connect a GitHub repository to import migration
files, or describe the desired schema in natural language for AI-generated
DDL. Regardless of the source path, all subsequent analysis runs against
the shadow instance.

\subsection{MCP Server Integration}

DB-Lighthouse exposes a Model Context Protocol (MCP) server at
\texttt{POST /api/mcp}, allowing external AI assistants (Claude Desktop,
Cursor) to call four DB-Lighthouse tools: \texttt{db\_query},
\texttt{db\_schema}, \texttt{db\_anomaly}, and \texttt{db\_optimize}. This
makes DB-Lighthouse an analytical backend for any MCP-compatible AI
frontend without code changes to either side.

\subsection{Deployment Model}

The platform is packaged as a three-service Docker Compose stack: the
Next.js frontend container, the FastAPI backend container, and the shadow
PostgreSQL container (\texttt{postgres:15-alpine}, health-gated via
\texttt{pg\_isready}). A single \texttt{docker-compose up --build}
invocation starts the full stack with no external accounts, API keys, or
managed services required. Three deployment tiers are supported without
architectural changes. \textbf{Local} deployment runs entirely on a
developer workstation. \textbf{Shared team} deployment runs the same image
on any container host (Railway, Render, Fly.io) with environment variables
pointing at the shadow database and AI endpoints. \textbf{Enterprise
on-premise} deployment satisfies data-residency requirements by running
entirely within a corporate network, relying on no mandatory cloud
dependencies; this tier additionally supports SAML-based SSO via the RBAC
configuration. The AI module defaults to Jan AI (\texttt{mistral-ins-7b-q4})
via an OpenAI-compatible REST interface, with fallback to Ollama or direct
GGUF loading via \texttt{llama-cpp-python} for fully air-gapped operation.

\section{Analytical Subsystem Architecture}

DB-Lighthouse organizes its analytical functionality into six architectural
subsystems, each encapsulating a distinct observational concern over the
shadow database. This section describes the data flows, algorithms, and
design decisions within each subsystem.

\subsection{Schema Introspection and Health Scoring}

Upon shadow database connection, the schema analysis service constructs a
directed metadata graph $G = (V, E)$ by querying \texttt{information\_schema}
and \texttt{pg\_catalog}. Each vertex $v \in V$ represents a table,
annotated with column definitions, data types, nullable flags, primary key
designations, estimated row counts from \texttt{pg\_class.reltuples}, and
storage sizes from \texttt{pg\_relation\_size}. Each directed edge
$e \in E$ represents a foreign key reference, weighted by the count of FK
relationships between the table pair. The visualization layer maps $G$ to
an interactive rendering via React Flow and D3-force; Dagre provides a
hierarchical layout for schemas with fewer than 30 tables, beyond which
D3-force repulsion takes precedence to avoid edge crossings.

A scalar health score $H$ aggregates findings across all analytical
subsystems:

\begin{equation}
    H = \max\!\left(0,\, \min\!\left(100,\; 100 - 15N_H - 8N_M - 3N_L\right)\right)
\end{equation}

where $N_H$, $N_M$, $N_L$ denote HIGH, MEDIUM, and LOW severity finding
counts. The asymmetric penalty weights ensure a single HIGH-severity
finding reduces $H$ below 85. AI-generated remediation advice is
parallelised across all findings via \texttt{asyncio.gather()}.

\subsection{Three-Phase Performance Optimization Engine}

\subsubsection{Index Analyzer}

The index analyzer executes three sequential phases against
\texttt{pg\_stat\_user\_tables} and \texttt{pg\_stat\_user\_indexes}.

\textbf{Phase A (Bottleneck Detection)} joins \texttt{seq\_scan} and
\texttt{n\_live\_tup} to identify tables with sequential scan counts
exceeding 100 and live row counts exceeding 10,000. The dual threshold
suppresses false positives from small, fully-cached tables while ensuring
sequential-scan cost is non-trivial.

\textbf{Phase B (Zombie Index Detection)} identifies entries in
\texttt{pg\_stat\_user\_indexes} with \texttt{idx\_scan = 0} that are
neither primary key nor unique constraints, as verified against
\texttt{pg\_constraint}. Such indexes impose write-path overhead with no
read benefit.

\textbf{Phase C (Missing Index Inference)} applies three heuristic rules:
(1) foreign-key columns lacking a supporting index, identified via an
\texttt{information\_schema} join; (2) tables with more than 100,000 live
rows exhibiting persistently high sequential-scan ratios; and (3) columns
with Pearson correlation $|r| < 0.1$ in \texttt{pg\_stats.correlation},
where heap ordering provides negligible benefit to range scans. Each
candidate receives a composite priority score:

\begin{equation}
    \text{score} = \frac{\text{raw\_score}}{\text{max\_score}} \times 80 + \text{risk\_bonus}
\end{equation}

where \texttt{risk\_bonus} $\in \{18, 12, 6\}$ encodes high, medium, and
low operational risk. Each recommendation is augmented with an
LLM-generated rationale grounded in the table statistics that triggered it.

\subsubsection{Storage Optimization Engine}

The storage engine computes per-column value-length statistics over a
sample of up to $N = 1{,}000$ rows:

\begin{equation}
    \mu_l = \frac{1}{N}\sum_{i=1}^{N} \text{len}(v_i),\quad
    \sigma_l = \sqrt{\frac{1}{N}\sum_{i=1}^{N}(\text{len}(v_i)-\mu_l)^2}
\end{equation}

Four heuristic rules derive candidates: \texttt{TEXT} columns with
$\mu_l < 255$ are flagged for \texttt{VARCHAR} narrowing; nullable columns
with a null rate of 0\% are candidates for \texttt{NOT NULL} enforcement;
columns with cardinality $\leq 20$ are candidates for enum normalization;
and indexes with \texttt{idx\_scan = 0} are flagged for removal. Each
recommendation carries a severity classification, an estimated byte saving,
and a complete DDL patch.

\subsubsection{Query Cost Estimation}

Every query executed against the shadow instance is instrumented with
\texttt{EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)}. The planner's total cost
node is mapped to estimated I/O volume and thence to monetary and
environmental cost metrics via the AWS Athena pricing model:

\begin{equation}
    b = \max\!\left(1,\, \left\lfloor\frac{\text{total\_cost}}{10}\right\rfloor\right) \times 8{,}192\ \text{bytes}
\end{equation}
\begin{equation}
    \text{cost}_{USD} = \frac{b}{2^{30}} \times \$0.10, \quad
    \text{CO}_2 = \frac{b}{2^{30}} \times 0.3\ \text{g}
\end{equation}

Chart type is inferred from the SELECT column signature: a two-column
result whose first column name contains \texttt{date}, \texttt{time}, or
\texttt{year} and whose second column is numeric produces a time-series
rendering; all other two-column numeric results produce a bar chart.

\subsection{Schema Governance: DDL Sandboxing, Versioning, and Drift}

\subsubsection{Transactional DDL Sandbox}

DDL statements submitted to \texttt{POST /api/lab/experiment} are executed
within an explicit transaction against the shadow instance using a Monaco
Editor interface with PostgreSQL syntax highlighting. On exception, the
transaction is rolled back and the complete PostgreSQL diagnostic is
returned. On success, the endpoint computes a structural diff by comparing
pre- and post-execution schema metadata---column tuples, index definitions,
and constraint sets---and returns the diff alongside the execution result.
Alternatively, the LLM generates DDL from a natural-language description
using the schema-aware prompt path described in Section~IV-D, with
DDL-specific steering injected into the system prompt.

\subsubsection{Schema Snapshot Versioning}

A snapshot operation serialises the current shadow schema via
\texttt{pg\_dump --schema-only} and persists the DDL string with a
timestamp and caller-supplied label in the shadow instance's history
relation. The inter-snapshot diff is computed as a three-way set difference
over table names, per-table column definition tuples, and per-table index
definition tuples. The snapshot detail endpoint
(\texttt{/snapshots/\{id\}/graph}) restores stored DDL into a temporary
schema namespace and re-runs the full introspection pipeline, producing a
graph payload structurally identical to the live schema graph.

\subsubsection{Schema Drift Detection}

The drift detector computes a set difference between the live shadow schema
metadata and a stored baseline across three dimensions: the table name set,
the per-table column definition set, and the per-table index definition
set. The result is a three-class change log (additions, modifications,
deletions) returned as structured JSON. Detected drift events are forwarded
to the incident engine as GOVERNANCE-severity incidents.

\subsection{AI Query Interface: Self-Healing NL-to-SQL and Voice}

\subsubsection{Self-Healing NL-to-SQL}

Each natural-language query is assembled into a schema-aware prompt
comprising the query text, the full table and column metadata extracted
from the shadow schema, business rules injected from
\texttt{semantic\_rules.json} (Section~\ref{sec:semantic}), and a system
instruction directing the model to produce SQL followed by a
natural-language explanation. The assembled prompt passes through the safety
firewall (Section~III-D) before LLM inference. The model response is parsed
via a DOTALL regex matching \texttt{SELECT} statements delimited by
triple-backtick code fences or end-of-string anchors, to extract the SQL block, which is dry-run against the shadow
instance. On a PostgreSQL exception, the error message is appended as a
correction hint and the model is re-invoked. This self-correction loop
executes at most three rounds before escalating with an error response. The
default inference backend is Jan AI running the \texttt{mistral-ins-7b-q4}
model \cite{jiang2023mistral}; a single environment variable substitutes
any OpenAI-compatible endpoint without code changes.

\subsubsection{Three-Tier Degradation-Resistant Voice Pipeline}

The voice endpoint implements a three-tier execution pipeline to maintain
availability under AI backend failure. Tier~1 routes the transcribed query
through the self-healing NL-to-SQL path. On AI backend failure, Tier~2
activates \texttt{build\_keyword\_sql()}, a schema-aware rule engine with
support for 27 country name constants, table synonym resolution (e.g.,
`sales' $\to$ \texttt{Invoice}), semantic column aliasing (e.g., `email
address' $\to$ \texttt{Email}), and COUNT versus SELECT intent
classification. Tier~3 provides server-side speech-to-text transcription
via a public STT endpoint, decoupling the pipeline from browser network
policy constraints. Audio is captured via the Web Audio API; speech
synthesis uses the Web Speech API.

\subsubsection{Multi-Agent Deliberation Council}

Governance-sensitive queries are routed to \texttt{council\_service.py},
which orchestrates a three-agent sequential deliberation. The
\textbf{Architect} agent proposes a performance-optimised query or DDL
strategy. The \textbf{Guardian} agent reviews the proposal for security
risks, PII exposure, and data-loss hazards. The \textbf{Executive} agent
synthesises both outputs into a final response at temperature 0.1 to
minimise hallucination. The complete deliberation transcript is returned
alongside the final SQL in the \texttt{council\_transcript} response field.

\subsubsection{Semantic Business Rule Enforcement}
\label{sec:semantic}

Named business invariants are stored in
\texttt{backend/data/semantic\_rules.json} as name-definition pairs and
injected into the LLM system prompt at inference time, enabling the model
to interpret domain terminology absent from the schema without retraining
(e.g., `active user' $\equiv$ logged in within 7 days). Rule submission
triggers an LLM call to generate a SQL assertion or trigger definition
encoding the invariant; the generated SQL is validated by a dry-run
\texttt{EXPLAIN} against the shadow instance. Violations trigger a second
LLM call returning a DDL remediation patch. Rule compliance status is
persisted in the shadow database's rules relation and re-evaluated on
demand via \texttt{/api/rules}.

\subsection{Statistical Anomaly Detection and Incident Prioritization}

\subsubsection{Z-Score Anomaly Detector}

The anomaly detector snapshots five metric categories from the shadow
instance (Table~\ref{tab:anomaly}), stored in a per-database JSON file
keyed by \texttt{base64(connection\_string)}. After a minimum of three
observations, a rolling mean $\mu$ and standard deviation $\sigma$ are
maintained over a 7-day sliding window per metric. The Z-score for a new
observation $x$ is:

\begin{equation}
    Z = \frac{x - \mu}{\sigma}
\end{equation}

$Z \geq 2.0$ generates a WARNING event; $Z \geq 3.0$ generates a CRITICAL
event. Confidence bands $[\mu - 2\sigma,\, \mu + 2\sigma]$ are returned
for visualisation. The \texttt{\_infer\_root\_cause()} method maps each
triggered metric to a domain explanation via a 12-entry static lookup
(e.g., elevated dead-tuple count $\to$ `autovacuum lag').

\subsubsection{Composite Incident Severity Engine}

Six incident categories are evaluated against threshold criteria
(Table~\ref{tab:incidents}). Each detected incident receives a composite
severity score:

\begin{equation}
    \text{severity} = 0.50 \times \text{impact} + 0.30 \times \text{frequency} + 0.20 \times \text{latency}
\end{equation}

where \textit{impact} encodes operational consequence on a 0--100 scale,
\textit{frequency} counts recurrences within the rolling window, and
\textit{latency} captures time-domain urgency. Score thresholds map to
four operational tiers: CRITICAL ($\geq 80$), HIGH ($\geq 60$), MEDIUM
($\geq 40$), LOW ($< 40$). Incident records are retained for 30 days;
raw metric snapshots are pruned after 7 days to bound storage growth.

\subsection{Security, PII Detection, and Collaborative DDL Governance}

\subsubsection{Two-Layer PII Exposure Scanner}

The PII scanner applies a two-layer detection pipeline. The first layer
matches column names against a 16-term case-insensitive regex covering
\texttt{ssn}, \texttt{credit\_card}, \texttt{passport},
\texttt{date\_of\_birth}, \texttt{phone}, \texttt{email},
\texttt{address}, and \texttt{bank\_account}. Flagged columns are sampled
at five rows for value-pattern analysis. The second layer applies
format-specific patterns: RFC 5322-compliant email, SSN format
(\texttt{\textbackslash d\{3\}-\textbackslash d\{2\}-\textbackslash d\{4\}}),
Luhn-validated credit card numbers, and IPv4/IPv6 addresses. A confidence
score is computed as the ratio of matched samples to total samples.
Detected PII columns are classified by type and severity; the synthetic
data generator (Faker library) produces statistically realistic
replacements for each PII type, enabling compliant test fixture generation
for GDPR, SOC~2, and HIPAA contexts.

\subsubsection{Collaborative Schema Change Workflow}

The collaboration subsystem implements a pull-request-analogous DDL review
workflow. Proposed modifications are stored as pending payloads with a
three-state status field (\texttt{pending}, \texttt{approved},
\texttt{rejected}); approved payloads are executed through the transactional
DDL sandbox described in Section~IV-C. Collaboration state is persisted via
a dual-backend strategy: Firestore is used when Firebase credentials are
present; a local JSON store provides an identical API surface for air-gapped
deployments. Every state transition---invitation, submission, approval,
rejection---is appended to an immutable audit log exposed via a read-only
endpoint. The data access layer provides parameterised server-side
pagination for table browsing, parameterised \texttt{UPDATE} within
transactions for cell-level mutation, and bulk \texttt{INSERT} from
BOM-aware CSV payloads for test data seeding.

\subsection{AI Fine-Tuning: Domain Adaptation Pipeline}

The default Jan AI / Mistral 7B configuration can be replaced with a
domain-fine-tuned model for improved first-attempt NL-to-SQL accuracy
on narrow schema tasks. The fine-tuning pipeline targets
Qwen2.5-Coder-7B-Instruct \cite{qwen2024}, a 7.6B-parameter
code-specialised model pre-trained on 5.5 trillion tokens including
substantial SQL and database documentation.

The model is loaded in 4-bit NormalFloat quantization (NF4) via the
BitsAndBytes library through Unsloth \cite{han2024unsloth}, reducing GPU
footprint from approximately 28~GB (float16) to approximately 6~GB
(int4), enabling training on a Google Colab T4 (15~GB VRAM). LoRA
\cite{hu2022lora} is applied to the query, key, value, output, gate, up,
and down projection matrices across all attention and feed-forward layers.
Trainable parameter count is approximately 41 million (0.54\% of 7.6B
base parameters). The training corpus covers PostgreSQL NL-to-SQL pairs
(Spider \cite{yu2018spider} subset), manually authored schema analysis
pairs, and adversarial injection-resistance pairs. Optimiser: AdamW with
cosine learning rate schedule, peak rate 2e-4, 5\% warmup.

\subsection{Index Recommendation Accuracy}
\label{sec:index-acc}

Index recommendations were evaluated by comparing system-generated
suggestions against a baseline set of expert DBA-authored index choices
for the same four schemas. Precision was computed as:

\begin{equation}
    \text{Precision} = \frac{|R_{\text{sys}} \cap R_{\text{expert}}|}{|R_{\text{sys}}|}
\end{equation}

where $R_{\text{sys}}$ is the set of system-generated recommendations and
$R_{\text{expert}}$ is the expert baseline. The system achieved a mean
precision of 0.81 across all schemas. False positives were mostly
composite index suggestions for queries the DBA had deprioritised on
cost-of-write grounds, a consideration not currently modelled by the
heuristic engine.

\subsection{User Experience}

We ran informal usability testing with five participants from the main
target group (backend developers with 2--7 years of experience). All
five connected a test database and reached the optimization report without
assistance within 5~minutes. Four out of five rated the plain-English
explanations as more helpful than reading PostgreSQL documentation. Three
said the shadow database safety guarantee made them less apprehensive
about connecting a production database. All five found the Lab DDL
sandbox useful for testing \texttt{CREATE INDEX} statements before
scheduling a migration.

% ─────────────────────────────────────────────────────────────────────────────
\section{Discussion}

\subsection{Shadow Database Trade-offs}

The shadow-database pattern provides a strong data-isolation guarantee at
the cost of schema staleness. If the production schema changes after
cloning, analysis reflects the old structure. The \texttt{pg\_dump}
approach also does not capture runtime statistics from production
(\texttt{pg\_stat\_user\_tables} values reflect the shadow instance),
which limits Phase A bottleneck accuracy for recently promoted tables.
Automated synchronisation via PostgreSQL event triggers or logical
replication would resolve both issues at the cost of added operational
complexity.

\subsection{Model Limitations}

The fine-tuned model was trained on PostgreSQL syntax. Queries targeting
MySQL or SQL Server may generate PostgreSQL-specific functions
(\texttt{ILIKE}, \texttt{generate\_series}) incompatible with the target.
The \texttt{dialect\_converter.py} post-processor covers approximately
80\% of common function incompatibilities. The 4,096-token context window
of the Q4\_K\_M configuration is insufficient for schemas with more than
40 tables, requiring schema chunking that can miss cross-table join
opportunities. The LLM-generated impact estimates (e.g., ``65\% latency
reduction'') are heuristic rather than quantitative; the platform does not
yet execute \texttt{EXPLAIN ANALYZE} on shadow-populated data.

\subsection{Anomaly Detection Sensitivity}

The $Z \geq 2.0$ warning threshold yields one false positive per 22
observations under a normal distribution ($P = 0.0455$). For a database
collecting hourly snapshots over 7 days, this produces approximately 3.1
spurious warnings per week. Raising the threshold to $Z \geq 2.5$ reduces
this to 0.6 per week ($P = 0.0124$) at the cost of reduced sensitivity.
The current threshold favours recall over precision, consistent with the
safety-first design philosophy.

\subsection{Council Latency}

The council pipeline adds 5,200~ms mean latency from three sequential LLM
calls (Table~\ref{tab:latency}). This is acceptable for
governance-sensitive operations (DDL generation, migration planning) but
unsuitable for interactive query answering. The system routes requests to
the council only via the \texttt{/deliberate} endpoint; conversational
queries bypass it and call the AI service directly.

\subsection{Scalability Constraints}

The baseline JSON store for anomaly metrics is a bottleneck under
concurrent multi-database load. Replacing it with a time-series database
(e.g., TimescaleDB) would support concurrent metric collection. The
NullPool connection strategy makes the FastAPI backend stateless and
load-balancer-ready. The binding constraint for horizontal scaling is the
Ollama inference endpoint, which is single-threaded by default and
requires GPU-aware request queuing under concurrent load.

\subsection{Security Boundary}

The four-layer security stack (shadow isolation, SQL guardrail, prompt
firewall, migration safety engine) does not cover supply-chain attacks
against the Python dependency tree. Firebase authentication is currently
mocked in local development, so production multi-user deployments need
explicit authentication hardening before network exposure. The
regex-based prompt firewall is susceptible to Unicode normalisation and
homoglyph-based bypass; a neural classifier would provide more robust
coverage at the cost of additional inference latency per request.

\subsection{Limitations of Current Evaluation}

There are additional limitations to the current version. The storage
analysis has a maximum sample of $N = 1000$ rows per table. Skewed
column distributions can look cheap compared to a full scan. Timeline
snapshots are initiated manually; a developer who fails to take a snapshot
before a migration loses that reference point.

\subsection{Future Work}

Five directions remain for future work. First, PostgreSQL event-trigger-based
schema synchronisation to eliminate manual re-cloning. Second, LSTM-based
forecasting in the anomaly detector to predict metric violations before
they occur. Third, multi-dialect SQL corpus expansion (MySQL, SQL Server,
BigQuery) to broaden model dialect coverage. Fourth, a neural classifier
in the prompt firewall to handle Unicode normalisation and homoglyph
bypass attempts. Fifth, filling the shadow database with anonymised
synthetic data to enable real \texttt{EXPLAIN ANALYZE} execution and
replace heuristic approximations with empirical estimates.

% ─────────────────────────────────────────────────────────────────────────────
\section{Conclusion}

DB-Lighthouse demonstrates that a shadow-database architecture, combined
with a four-layer security stack and a domain-fine-tuned language model,
can deliver production-grade database intelligence without reading
production data. The shadow-database pattern enforces data isolation at
the infrastructure level. The prompt firewall and SQL guardrail intercept
adversarial inputs before model inference and after SQL generation
respectively. The two-agent council separates performance optimisation from
security review. A LoRA fine-tuning pipeline adapts a 7.6B-parameter
code model to database administration tasks within 15~GB VRAM and exports
a 4.6~GB GGUF model for commodity hardware deployment.

Measured on the Chinook benchmark: 100\% NL-to-SQL resolution within
three self-healing rounds (76\% first-attempt success); 100\% prompt
injection detection across 62 adversarial inputs with zero false positives;
97.5--98.4\% execution-time reduction for targeted point-lookup queries
after index simulation; and 98.7\% overall voice query accuracy across 75
spoken queries. All safety mechanisms add measurable but bounded latency:
the prompt firewall under 1~ms, migration safety 95~ms mean, and council
deliberation 5,200~ms mean for governance queries only. On four production
schemas, data type optimization and redundant index removal decreased
storage by a mean of 19.4\% and index recommendation precision was 0.81
compared to expert DBA choices.

The six analytical subsystems collectively span the full schema health
lifecycle, from introspection and optimization through governance,
security, and incident response, within a single shadow-bounded interface.
Six database back-ends---PostgreSQL, MongoDB, Neo4j, Firebase, MySQL, and
SQLite. These findings demonstrate that rule-based heuristics, Z-score
anomaly detection, catalog introspection, force-directed visualization,
DDL sandboxing, schema snapshotting, multi-agent council deliberation, a
self-healing NL-to-SQL loop, and a locally-hosted domain-fine-tuned LLM
are sufficient to create a database observability tool that a small team
can truly use---without spending money on enterprise DBA software or
exposing a single row of production data.
